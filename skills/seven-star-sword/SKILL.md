# 七星剑 — 七步结构化开发流程

一个需求驱动的七步开发闭环。每一步必须获得用户确认后才能进入下一步，不允许跳步。

## 流程总览

```
① 需求探索 → ② 需求对齐 → ③ 方案展示 → ④ 文档落盘 → ⑤ 执行任务 → ⑥ 审核改动 → ⑦ 注释提交
```

**条件分支**：当需求已经足够清晰（用户明确描述了要改什么、在哪改、预期效果），跳过 ① 直接进入 ② 需求对齐。

每一步结束时，向用户展示本步结论并明确询问"确认进入下一步？"，用户确认后才继续。

---

## 第一步：需求探索

**目标**：当需求模糊或只是一个想法时，通过协作对话将想法收敛为明确的需求描述。

**何时触发**：需求描述过于宽泛（如"优化通知"、"加个功能"）、只有方向没有细节、或是全新功能从零开始。

**何时跳过**：需求已经明确描述了目标、范围、预期效果，直接进入第二步。

### 1.1 探索项目上下文

根据需求关键词，用 GitNexus 查询相关执行流程和代码结构，把理解建立在代码事实之上：

- `gitnexus_query({query: "需求关键词"})` — 找相关执行流程
- `gitnexus_context({name: "符号名"})` — 查具体符号的上下文
- 查看最近 commit 了解项目当前状态

### 1.2 创建探索待办列表

将需要澄清的方向整理成 Todo 列表，逐个提问而非一次性抛出：

- 调用 `TodoWrite` 创建临时 todo 列表，每个澄清方向一个 todo：
  - `status: "pending"`，`priority: "medium"`
  - `content` 使用 `"探索: {澄清方向}"` 格式
- 将第一个 todo 标记为 `in_progress`，只向用户提出这一个问题
- 用户回答后，将该 todo 标记为 `completed`，下一个标记为 `in_progress`，继续提问
- 依次进行直到所有方向澄清完毕

### 1.3 提问原则

- **每次一个问题**，不要同时抛出多个问题
- **优先选择题**，在可能的情况下比开放式问题更容易回答
- 重点理解：目的、约束、成功标准
- 评估范围：如果需求涉及多个独立子系统，先帮用户拆分，再对第一个子项目进行探索

### 1.4 提出方案

需求方向明确后，提出 2-3 种不同的实现方案：

- 以对话方式展示选项，附上权衡分析和推荐理由
- 先展示推荐方案并解释原因
- 用户选择方案后，输出明确的需求描述

### 1.5 输出需求描述

收敛为一段清晰的需求描述，包含：

- **目标**：要解决什么问题或实现什么功能
- **范围**：涉及哪些模块/服务
- **预期效果**：改完之后应该是什么样子
- **约束**：技术限制、兼容性要求等

用户确认需求描述后，进入第二步。

> 如果用户说"需求就这样，别问了"，直接输出你理解的需求描述让他确认，不要反复追问。

---

## 第二步：需求对齐

**目标**：彻底理解用户要做什么，消除歧义。

1. **先理解，再对齐**。拿到需求后先用自己的话复述一遍，让用户确认理解是否正确。如果从第一步进入，复述第一步产出的需求描述。
2. **主动探索代码**。根据需求关键词，用 GitNexus 查询相关执行流程和代码结构，把理解建立在代码事实上，而不是假设上：
   - `gitnexus_query({query: "需求关键词"})` — 找相关执行流程
   - `gitnexus_context({name: "符号名"})` — 查具体符号的上下文
3. **创建对齐待办列表**。将所有需要对齐的歧义点整理成 Todo 列表，逐个提问而非一次性抛出：
   - 调用 `TodoWrite` 创建临时 todo 列表，每个歧义点一个 todo：
     - `status: "pending"`，`priority: "medium"`
     - `content` 使用 `"对齐: {歧义点描述}"` 格式
   - 将第一个 todo 标记为 `in_progress`，只向用户提出这一个问题
   - 用户回答后，将该 todo 标记为 `completed`，下一个标记为 `in_progress`，继续提问
   - 依次进行直到所有歧义点对齐完毕
4. **常见歧义点参考**（根据实际需求选取，不必全部包含）：
   - 影响范围（哪些模块/服务）
   - 输入输出格式
   - 边界条件（空值、异常、并发）
   - 与现有功能的交互方式
5. **确认后输出**：一段清晰的需求摘要，包含目标、范围、约束。用户确认后进入第三步。

> 如果用户说"需求就这样，别问了"，直接输出你理解的需求摘要让他确认，不要反复追问。

---

## 第三步：方案展示

**目标**：讲清楚怎么改，变动是什么，影响有多大。

1. **列出变更清单**。每个变更点说明：
   - 文件路径
   - 改什么（新增/修改/删除）
   - 改动原因
2. **运行 GitNexus 影响分析**（必须）。对方案涉及的每个核心符号运行：
   ```
   gitnexus_impact({target: "符号名", direction: "upstream"})
   ```
   向用户报告影响范围（直接调用方、受影响的执行流程、风险等级）。
3. **如果影响分析返回 HIGH 或 CRITICAL 风险，必须向用户明确警告**，由用户决定是否继续。
4. **说明不可逆操作**（删表、改接口签名等），确认用户知情。
5. 用户确认方案后进入第四步。

---

## 第四步：文档落盘

**目标**：将方案写成"可交接"的技术文档——换一个开发者拿到文档就能直接动手开发，不需要
再问问题、不需要额外看代码、不需要猜测意图。

### 4.1 确定文档路径

- 目录：`docs/plan/YYYY-MM-DD-{需求概要}/`
- 文件：`{需求概要}.md`
- 需求概要用简短中文，如 `欠费通知重构`、`页面通知始终发送`。

### 4.2 代码核实（写入前必须）

在落盘文档前，必须对每个改动点做代码核实。这一步是"可交接"标准的基石——
文档里的代码片段如果和实际文件不一致，执行者就会写出错误的代码。

1. **读取每个要修改的文件**，确认：
   - 方法签名与文档描述一致（参数类型、返回值、修饰符）
   - 行号准确（代码可能在上次读取后已变更）
   - 改前代码片段与文件内容逐行一致
2. **读取每个被调用的方法**（特别是新增调用的方法），确认：
   - 参数签名——能否传 null、参数顺序
   - 方法内部是否有守卫逻辑（如 null 检查、前置条件）会影响改动
   - 方法实际行为是否与文档假设一致
3. **如发现文档描述与代码不一致**，修正文档后再落盘，不要假设"差不多就行"。
   常见不一致：方法签名不同、null 守卫导致传 null 静默失败、实际逻辑是双路径但文档写成单路径。

### 4.3 向用户展示预览后落盘

写入前先展示完整文档内容供用户确认，确认后再落盘。

### 4.4 文档必须包含的内容（"可交接"标准）

文档必须以 Markdown 格式编写，包含以下章节。每节的写作标准：

**标题**：`# {简短描述} — {一句话补充说明}`

**## 日期**：YYYY-MM-DD

**## 背景**：说清楚三件事——

- **当前是什么样子**（现状描述，包括涉及的关键类、方法、执行流程）
- **有什么问题**（为什么需要改）
- **影响入口**（触发这个逻辑的调用入口有哪些）

**## 决策**：用一两句话概括要改什么、怎么改。这是给快速阅读的人看的摘要。

**## 当前代码结构**：用 ASCII 框图和表格说清楚——

- 类关系（谁依赖谁，新增/已有标注清楚）
- 完整调用链路（从入口到出口，标注每个改动点）
- 关键域对象（涉及的数据表/实体，列出关键字段及其含义）
- 枚举/常量（如果改动涉及枚举值，列出全部值及其含义）

**## 变更内容**：每个文件一小节，每节包含——

- **文件路径**（完整路径，从项目根开始）
- **具体改动**：位置（行号或方法名）+ 改前代码片段 + 改后代码片段。代码块用 Java 语法高亮。
- 对于替换整个方法的情况：展示完整的新方法体。
- 对于单行/几行的改动：用 diff 式对比，标注"改前"/"改后"。

**## 任务拆解**：表格，每行一个任务——

| 编号 | 任务 | 文件 | 具体步骤 |

- 编号按执行顺序排列
- 具体步骤用 ① ② ③ 列举，让执行者可以逐个打勾
- 如果任务间有依赖关系，在表格后注明
- 第五步执行时，每完成一个任务会在「编号」列前追加 `✅` 标记，并在「具体步骤」列每个步骤编号后追加 `✅`

**## 影响范围**：说明改动会影响哪些调用链路、哪些模块。

- 直接调用方（表格：符号、影响说明、风险）
- 间接影响（通过调用链路传递）

**## 注意事项**：列出执行时必须注意的点——

- 会不会重复执行（如通知发两次）
- null 安全性（传 null 是否安全，解释原因）
- 边界情况（空 noticeId、空配置等）
- 日志变更（如果改了日志关键字，说明原因）
- 参数顺序（如果构造函数参数顺序重要）

**## 不做事项**：明确列出本次不改什么，防止执行者做多。

### 4.5 内容格式规范

- 代码块始终用 ` ```java ` 包裹
- ASCII 框图用 ` ``` ` 包裹
- 表格按"人阅读优先"的列顺序排列，不是"机器执行优先"
- 不要用缩写路径，始终写完整路径

---

## 第五步：执行任务

**目标**：按文档执行，以文档为准。

> 第五步开始时会创建 Todo 列表展示所有子任务进度。每完成一个子任务，Todo 和文档中的「任务拆解」表格会同步更新完成标记。

1. **重新阅读文档**。文档可能在落盘后被用户修改过，执行前必须重新读取确认。
2. **创建 Todo 列表**。调用 `TodoWrite` 将文档「任务拆解」表格中所有子任务同步为 todo：
   - `status: "pending"`，`priority: "high"`
   - `content` 使用 `"T{N}: {任务描述} — {文件}"` 格式
   - 全部同步后，告知用户"共 N 个子任务，开始执行"。
3. **开始执行子任务**。将要执行的子任务在 `TodoWrite` 中标记为 `in_progress`，向用户输出"开始执行 T{N}: {任务描述}"。
4. **完成子任务并更新文档**。子任务执行完毕后，同时做两件事：
   - ① 调用 `TodoWrite` 将该子任务状态更新为 `completed`，下一个子任务标记为 `in_progress`
   - ② 调用 `Edit` 更新文档的「任务拆解」表格——在对应行的「编号」列追加 `✅`（如 `\| T0 \|` → `\| ✅ T0 \|`），并在「具体步骤」列每个步骤编号后追加 `✅`（如 `① 读取文件` → `①✅ 读取文件`）
   - 完成后向用户输出"T{N} 完成 ✅"。
5. **按任务拆解顺序执行**，重复步骤 3-4 直到所有子任务完成。
6. **遇到文档未覆盖的情况**，暂停并向用户确认，不要自行决定。
7. **遵循项目编码规范**：
   - 后端：MyBatis-Plus 查询不需要手动加 `TenantHelper.dynamic` 包装
   - 前端：遵循 `front/AGENTS.md` 中的规范
   - 不删除注释掉的代码（可能暂时有用）
   - 简单三元判断不换行
8. 所有子任务完成后，向用户展示完成摘要（已完成数/总数），进入第六步。

---

## 第六步：审核改动

**目标**：验证改动是否与需求对齐，是否真的完成，代码质量是否达标。

> 第六步开始时会创建 Todo 列表展示所有审核项进度。每完成一项审核，Todo 会同步更新。

### 6.1 运行变更检测

```
gitnexus_detect_changes({scope: "unstaged"})
```

查看实际改动涉及的符号和执行流程，与文档预期对比。

### 6.2 需求完整性审核（逐项打勾）

创建 Todo 列表，逐项检查每个需求点：

- [ ] **需求点 N**：{需求描述} — 读取对应代码，确认实现与文档一致
- 检查位置：方法名、行号、参数、返回值
- 检查逻辑：条件判断、异常处理、日志记录
- 如发现问题，修复后重新检查该项

### 6.3 变更范围审核（逐项打勾）

- [ ] **文件范围**：只修改了文档中列出的文件，无额外改动
- [ ] **符号范围**：只改动了文档中列出的符号，无额外改动
- [ ] **执行流程**：未影响文档未提及的执行流程

### 6.4 代码质量审核（逐项打勾）

这是第六步的核心，必须逐项检查：

#### 可读性
- [ ] **命名清晰**：变量名、方法名是否清晰表达意图
- [ ] **逻辑简洁**：是否有不必要的嵌套、复杂条件判断
- [ ] **一致性**：与项目现有代码风格是否一致

#### 注释
- [ ] **新增方法**：是否有 Javadoc 说明用途、参数、返回值
- [ ] **复杂逻辑**：关键步骤是否有行内注释
- [ ] **注释密度**：与项目现有注释密度是否一致（不过度、不缺失）
- [ ] **注释准确**：注释是否与代码实际行为一致

#### 过度封装
- [ ] **方法粒度**：方法是否过长（超过50行需警惕）或过短（只有一行的私有方法）
- [ ] **抽象层次**：是否有不必要的抽象层、接口、设计模式
- [ ] **间接调用**：调用链是否过深（超过3层需警惕）

#### 性能
- [ ] **数据库查询**：是否有 N+1 查询、缺少索引、全表扫描
- [ ] **循环内操作**：循环内是否有数据库调用、网络请求、重量级计算
- [ ] **并发安全**：是否有共享变量、线程安全问题
- [ ] **资源释放**：是否有未关闭的资源（流、连接、线程池）
- [ ] **缓存使用**：是否合理使用缓存，避免重复计算/查询

### 6.5 审核结果

- 如发现问题，修复后重新审核对应项
- 所有审核项通过后，向用户展示审核结果摘要
- 用户确认审核通过后进入第七步

---

## 第七步：注释提交

**目标**：写好注释，提交代码和文档。

1. **补充注释**：
   - 新增的类/方法添加 Javadoc 或行内注释说明用途
   - 复杂逻辑添加关键步骤注释
   - 不要过度注释，保持与项目现有注释密度一致
2. **提交前再次运行变更检测**（AGENTS.md 要求）：
   ```
   gitnexus_detect_changes({scope: "compare", base_ref: "master"})
   ```
   确认改动范围符合预期。
3. **提交代码**：`feat/fix/refactor/docs: 简要描述本次改动`
4. **提交计划文档**：`docs: 补充{需求概要}计划文档`
   - 文档路径：`docs/plan/YYYY-MM-DD-{需求概要}/{需求概要}.md`
   - 确保文档中的任务拆解表格已更新完成标记（✅）
5. 流程结束。

---

## 异常处理

- **用户要求回退**：如果在第 5/6 步发现方案有问题，回到第 3 步重新对齐方案，更新文档后继续。
- **用户要求终止**：随时可以终止，已完成的工作成果（特别是文档）保留在 `docs/plan/` 中。
- **步骤跳转**：正常流程不允许跳步，但用户可以明确要求回到某个步骤重新来过。
