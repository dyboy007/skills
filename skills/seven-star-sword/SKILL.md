# 七星剑 — 七步结构化开发流程

一个需求驱动的七步开发闭环。每一步必须获得用户确认后才能进入下一步，不允许跳步。

## 流程总览

```
① 需求探索 → ② 需求对齐 → ③ 方案展示 → ④ 文档落盘 → ⑤ 执行任务 → ⑥ 审核改动 → ⑦ 注释提交
```

**条件分支**：
- 需求已经足够清晰（用户明确描述了要改什么、在哪改、预期效果）→ 跳过 ①，从 ② 需求对齐开始
- 用户提供了完整设计文档（包含变更清单、代码片段、任务拆解）→ 跳过 ①②③，从 ④ 文档落盘开始（但需先核实文档与代码的一致性）

每一步结束时，向用户展示本步结论并明确询问"确认进入下一步？"，用户确认后才继续。

## 适用边界

**适用**：涉及多文件改动、需要设计方案、影响多个模块的开发任务。
**不适用**：
- 单行/单文件 bug 修复
- 纯配置变更（YAML、SQL、环境变量）
- 纯文档编写
- 已有详细设计文档的执行任务（直接进第五步）

**快速判断**：如果改完不需要写文档就能交接，就不需要七星剑。

## 领域术语（CONTEXT.md）

项目根目录的 `CONTEXT.md` 是领域术语表，贯穿整个流程。

- **读**：遇到不熟悉的术语，先查 CONTEXT.md。有定义就直接采用，不问用户。
- **验**：CONTEXT.md 没有的术语，用户解释后，去代码中验证定义是否与实际行为一致。矛盾时指出来。
- **写**：确认后的术语写入 CONTEXT.md，格式如下。只收录项目特有的领域术语，通用概念不收录。

```md
**{术语名}**:
{一两句话定义，说清楚它是什么}
_Avoid_（避免使用）: {同义词或易混淆词}
```

## GitNexus 降级策略

当 GitNexus 不可用时，按以下方式降级，不影响流程推进：

| 原操作 | 降级方案 |
|--------|----------|
| 探索/对齐（query/context） | Grep + Read 手动追踪 |
| 影响分析（impact） | grep 追踪调用链，文档注明"手动分析，可能不完整" |
| 变更检测（detect_changes） | `git diff` 替代 |

遇到 GitNexus 不可用时，先尝试 `gitnexus analyze` 更新索引；仍不可用则按上表降级。

## 第一步：需求探索

**目标**：当需求模糊或只是一个想法时，通过协作对话将想法收敛为明确的需求描述。

**何时触发**：需求描述过于宽泛（如"优化通知"、"加个功能"）、只有方向没有细节、或是全新功能从零开始。

**何时跳过**：需求已经明确描述了目标、范围、预期效果，直接进入第二步。

### 1.1 术语预读 + 上下文探索

1. 读 CONTEXT.md，识别需求中涉及的领域术语（有定义的直接采用，没有的记录待对齐）
2. 用 GitNexus 查询相关执行流程和代码结构：
   - `gitnexus_query({query: "需求关键词"})` — 找相关执行流程
   - `gitnexus_context({name: "符号名"})` — 查具体符号的上下文
3. 查看最近 commit 了解项目当前状态

### 1.2 逐个澄清

将需要澄清的方向整理成 Todo 列表，逐个提问：

- 调用 `TodoWrite` 创建临时 todo 列表，每个澄清方向一个 todo：
  - `status: "pending"`，`priority: "medium"`
  - `content` 使用 `"探索: {澄清方向}"` 格式
- 将第一个 todo 标记为 `in_progress`，只向用户提出这一个问题
- 用户回答后，将该 todo 标记为 `completed`，下一个标记为 `in_progress`
- 探索方向可动态变化：用户回答一个问题后可能引出新问题，随时追加
- 依次进行直到所有方向澄清完毕

**提问原则**：
- **每次一个问题**，不要同时抛出多个
- **优先选择题**，比开放式问题更容易回答
- **术语精化**：发现模糊或重载术语（如"通知"可能指消息推送也可能指页面弹窗），要求用户精确定义
- **场景压力测试**：用具体场景探查边界，例如"如果用户同时收到两条欠费通知会怎样？"
- **交叉验证**：用户描述现有行为时，用代码验证。发现矛盾时指出："代码里是 X，但你说的是 Y——哪个对？"
- 评估范围：涉及多个独立子系统时，先帮用户拆分，再对第一个子项目探索

### 1.3 规模评估

根据探索结果评估需求规模并告知用户：

| 规模 | 文件数 | 影响深度 | 文档模板 | 建议策略 |
|------|--------|----------|----------|----------|
| 小 | ≤ 3 | 单模块，调用链 ≤ 2 层 | 轻量版 | 一口气走完 |
| 中 | 4-10 | 跨模块，调用链 2-3 层 | 标准版 | 正常七步流程 |
| 大 | > 10 | 跨服务，调用链 > 3 层 | 完整版 + 分阶段 | 拆分为多个独立子任务 |

影响深度判断依据：第三步影响分析的风险等级（LOW/MEDIUM/HIGH）可辅助确认规模。

大规模需求必须拆分，拆分原则：
- 按模块/服务边界拆分（如后端改动 vs 前端改动）
- 按执行顺序拆分（如先改数据库 → 再改后端 → 最后改前端）
- 每个子任务独立可验证

### 1.4 提出方案 + 输出需求描述

需求方向明确后，提出 2-3 种实现方案（附权衡分析和推荐理由），用户选择后输出需求描述：

- **目标**：要解决什么问题或实现什么功能
- **范围**：涉及哪些模块/服务
- **预期效果**：改完之后应该是什么样子
- **约束**：技术限制、兼容性要求等

同时，将本次确认的术语写入 CONTEXT.md（如果尚未收录）。

用户确认需求描述后，进入第二步。

> 如果用户说"需求就这样，别问了"，直接输出你理解的需求描述让他确认，不要反复追问。

## 第二步：需求对齐

**目标**：彻底理解用户要做什么，消除歧义。

1. **复述需求**。用自己的话复述需求描述，让用户确认理解是否正确。如果从第一步进入，复述第一步的产出。
2. **代码探索**。根据需求关键词，用 GitNexus 查询相关执行流程和代码结构，把理解建立在代码事实上：
   - `gitnexus_query({query: "需求关键词"})` — 找相关执行流程
   - `gitnexus_context({name: "符号名"})` — 查具体符号的上下文
3. **记录探索结果**。调用 `TodoWrite` 创建一个 `"探索摘要"` todo，记录关键事实（如"这个方法有 3 个调用方"、"该逻辑走双路径"）。这些事实会写入第四步文档的「背景」章节。
4. **逐个对齐歧义点**。将歧义点整理成 Todo 列表，逐个提问：
   - `content` 使用 `"对齐: {歧义点描述}"` 格式
   - 同第一步的提问原则（每次一个、优先选择题、场景压力测试、交叉验证）
5. **常见歧义点参考**（根据实际需求选取）：
   - 影响范围（哪些模块/服务）
   - 输入输出格式
   - 边界条件（空值、异常、并发）
   - 与现有功能的交互方式
6. **确认后输出**：一段清晰的需求摘要，包含目标、范围、约束。用户确认后进入第三步。

> 如果用户说"需求就这样，别问了"，直接输出你理解的需求摘要让他确认，不要反复追问。

---

## 第三步：方案展示

**目标**：讲清楚怎么改，变动是什么，影响有多大。

1. **列出变更清单**。每个变更点说明：
   - 文件路径
   - 改什么（新增/修改/删除）
   - 改动原因
2. **运行 GitNexus 影响分析**（必须）。对方案涉及的每个核心符号运行：
   ```
   gitnexus_impact({target: "符号名", direction: "upstream"})
   ```
   向用户报告影响范围（直接调用方、受影响的执行流程、风险等级）。
3. **如果影响分析返回 HIGH 或 CRITICAL 风险，必须向用户明确警告**，由用户决定是否继续。
4. **说明不可逆操作**（删表、改接口签名等），确认用户知情。
5. 用户确认方案后进入第四步。

---

## 第四步：文档落盘

**目标**：将方案写成"可交接"的技术文档——换一个开发者拿到文档就能直接动手开发，不需要再问问题、不需要额外看代码、不需要猜测意图。

### 4.1 确定文档路径

- 目录：`docs/plan/YYYY-MM-DD-{需求概要}/`
- 主文档：`{需求概要}.md`
- SQL 文件（如有）：`{表名}.sql`，每个涉及的表一个文件
- 需求概要用简短中文，如 `扣费重算重构`、`告警检测功能实现`。

### 4.2 代码核实（写入前必须）

在落盘文档前，必须对每个改动点做代码核实。文档里的代码片段如果和实际文件不一致，执行者就会写出错误的代码。

1. **读取每个要修改的文件**，确认：
   - 方法签名与文档描述一致（参数类型、返回值、修饰符）
   - 行号准确（代码可能在上次读取后已变更）
   - 改前代码片段与文件内容逐行一致
2. **读取每个被调用的方法**（特别是新增调用的方法），确认：
   - 参数签名——能否传 null、参数顺序
   - 方法内部是否有守卫逻辑（如 null 检查、前置条件）
   - 方法实际行为是否与文档假设一致
3. **如发现不一致**，修正文档后再落盘，不要假设"差不多就行"。

#### 编译验证（可选但推荐）

后端改动落盘前执行编译验证：

```bash
mvn clean compile -pl rx-application/rx-admin-server/rx-admin-saas -am -Plocal
```

如时间紧迫可跳过，但**必须在文档开头注明"未经编译验证"**。

### 4.3 预览后落盘

- 文档较短（< 100 行）：直接展示全文
- 文档较长（≥ 100 行）：展示目录结构 + 关键章节摘要，告知用户"完整文档已准备好，确认后落盘"
- 用户确认后调用 Write 落盘

### 4.4 文档模板

根据第一步的规模评估选择模板深度。所有代码块用 ` ```java ` 包裹，ASCII 框图用 ` ``` ` 包裹，路径写完整不用缩写。

#### 轻量版（小规模：≤ 3 文件，单模块）

```md
# {简短描述} — {一句话补充说明}

## 日期
YYYY-MM-DD

## 背景
当前是什么样子、有什么问题、影响入口。

## 决策
一两句话概括要改什么、怎么改。

## 变更内容
每个文件一小节：
- **文件路径**
- **具体改动**：位置（行号或方法名）+ 改前代码片段 + 改后代码片段

## 任务拆解
| 编号 | 任务 | 文件 | 依赖 | 具体步骤 |
编号按执行顺序排列，依赖列标注前置任务编号（无依赖填 `—`），步骤用 ① ② ③ 列举。

## 验证方式
- 后端：编译命令 / 单元测试
- 前端：lint 命令 / 手动验证步骤
```

#### 标准版（中规模：4-10 文件，跨模块）

在轻量版基础上增加：

```md
## 当前代码结构
用 ASCII 框图和表格说明：
- 类关系（谁依赖谁，新增/已有标注清楚）
- 完整调用链路（从入口到出口，标注每个改动点）
- 关键域对象（涉及的数据表/实体，列出关键字段）

## 影响范围
- 直接调用方（表格：符号、影响说明、风险）
- 间接影响（通过调用链路传递）

## 注意事项
- null 安全性、边界情况、参数顺序等执行时必须注意的点

## 不做事项
明确列出本次不改什么，防止执行者做多。
```

#### 完整版（大规模：> 10 文件，跨服务）

在标准版基础上增加：

```md
## SQL 变更
- SQL 文件独立存放：`docs/plan/YYYY-MM-DD-{需求概要}/{表名}.sql`
- 文件开头用 SQL 注释说明变更原因和关联需求
- 文档中记录：表名、文件路径、执行顺序（相对于代码变更）、回滚方式

## 配置变更（如有）
- 文件路径、变更内容（改前/改后）、是否需要重启服务、是否需要同步到其他环境

## 枚举/常量
如果改动涉及枚举值，列出全部值及其含义。

## 需求验证矩阵
每个需求点对应一个验证方式，执行时逐项打勾确认。
```

### 4.5 文档版本管理

- 文档落盘后如需修改，由用户手动编辑或在第六步审核时由 agent 建议修改
- 执行过程中只更新「任务拆解」表格的完成标记，不修改其他章节
- 如发现文档与代码不一致，暂停执行并告知用户

## 第五步：执行任务

**目标**：按文档执行，以文档为准。

> 第五步开始时会创建 Todo 列表展示所有子任务进度。每完成一个子任务，Todo 和文档中的「任务拆解」表格会同步更新完成标记。

1. **重新阅读文档**。以下情况必须重新读取：
   - 新会话继续执行（上下文可能丢失）
   - 用户明确说"文档已修改"
   - 距离落盘超过 30 分钟
   
   同一会话内连续执行可跳过此步。
2. **创建 Todo 列表**。调用 `TodoWrite` 将文档「任务拆解」表格中所有子任务同步为 todo：
   - `status: "pending"`，`priority: "high"`
   - `content` 使用 `"T{N}: {任务描述} — {文件}"` 格式
   - 全部同步后，告知用户"共 N 个子任务，开始执行"。
3. **开始执行子任务**。将要执行的子任务在 `TodoWrite` 中标记为 `in_progress`，向用户输出"开始执行 T{N}: {任务描述}"。
   - 优先执行无依赖的任务，有依赖的任务等前置任务完成后自动解锁
4. **完成子任务并更新文档**。子任务执行完毕后，同时做两件事：
   - ① 调用 `TodoWrite` 将该子任务状态更新为 `completed`，下一个子任务标记为 `in_progress`
   - ② 调用 `Edit` 更新文档的「任务拆解」表格——在对应行的「编号」列前追加 `✅`（如 `| T0 |` → `| ✅ T0 |`），并在「具体步骤」列每个步骤编号后追加 `✅`（如 `① 读取文件` → `①✅ 读取文件`）
   - 完成后向用户输出"T{N} 完成 ✅"。
5. **按任务拆解顺序执行**（遵循依赖关系），重复步骤 3-4 直到所有子任务完成。
6. **遇到文档未覆盖的情况**，暂停并向用户确认，不要自行决定。
7. **遵循项目编码规范**：
   - 后端：MyBatis-Plus 查询不需要手动加 `TenantHelper.dynamic` 包装
   - 前端：遵循 `front/AGENTS.md` 中的规范
   - 不删除注释掉的代码（可能暂时有用）
   - 简单三元判断不换行
8. 所有子任务完成后，向用户展示完成摘要（已完成数/总数），进入第六步。

### 执行中断与恢复

**中断场景**及处理方式：

| 场景 | 处理 |
|------|------|
| 用户要求暂停 | 记录当前进度（已完成的 todo + 文档中的 ✅ 标记），告知用户"下次可直接继续" |
| 用户要求终止 | 保留已完成的工作成果（特别是文档），流程结束 |
| 发现方案有问题 | 暂停执行，回到第三步重新对齐方案，更新文档后继续 |
| 子任务执行失败 | 记录失败原因，暂停执行，向用户展示失败任务和已完成任务的状态 |
| 代码冲突 | 暂停执行，向用户展示冲突文件，由用户解决后继续 |
| 需求变更 | 评估影响范围：小改直接更新文档继续，大改回到第二步重新对齐 |

**恢复方式**：新会话继续时：
1. 读取文档中的「任务拆解」表格
2. 对已完成的（带 ✅）任务做一次快速 diff 检查，确认代码未被他人覆盖
3. 从未完成的任务继续执行

## 第六步：审核改动

**目标**：验证改动是否与需求对齐，是否真的完成，代码质量是否达标。

> 第六步开始时会创建 Todo 列表展示所有审核项进度。每完成一项审核，Todo 会同步更新。

### 6.1 运行变更检测

```
gitnexus_detect_changes({scope: "unstaged"})
```

查看实际改动涉及的符号和执行流程，与文档预期对比。

### 6.2 需求完整性审核（逐项打勾）

创建 Todo 列表，逐项检查每个需求点：

- [ ] **需求点 N**：{需求描述} — 读取对应代码，确认实现与文档一致
- 检查位置：方法名、行号、参数、返回值
- 检查逻辑：条件判断、异常处理、日志记录
- 如发现问题，修复后重新检查该项

### 6.3 功能验证

根据文档中的「验证方式」章节执行验证：

- **后端改动**：
  - 编译通过：`mvn clean compile -pl {模块} -am -Plocal`
  - 如有单元测试，运行并确认通过
  - 关键路径：检查日志输出是否符合预期
- **前端改动**：
  - lint 通过：`pnpm lint:eslint`
  - 手动验证：按文档中的验证步骤在浏览器中确认 UI 变更
- **跨服务改动**：
  - 在 dev 环境联调，确认端到端流程正常

### 6.4 变更范围审核（逐项打勾）

- [ ] **文件范围**：只修改了文档中列出的文件，无额外改动
- [ ] **符号范围**：只改动了文档中列出的符号，无额外改动
- [ ] **执行流程**：未影响文档未提及的执行流程

### 6.5 代码质量审核

审核按风险等级分层。根据第三步影响分析的风险等级，选择对应的审核深度：

#### 必检项（所有改动必查）

- [ ] **命名清晰**：变量名、方法名是否清晰表达意图
- [ ] **逻辑简洁**：是否有不必要的嵌套、复杂条件判断
- [ ] **一致性**：与项目现有代码风格是否一致
- [ ] **null 安全**：新增的方法调用是否处理了 null 情况
- [ ] **异常处理**：关键路径是否有 try-catch，日志是否包含足够上下文
- [ ] **注释准确**：新增注释是否与代码实际行为一致

#### 推荐项（LOW 风险可抽查，MEDIUM/HIGH 风险必查）

- [ ] **新增方法**：是否有 Javadoc 说明用途、参数、返回值
- [ ] **注释密度**：与项目现有注释密度是否一致（不过度、不缺失）
- [ ] **方法粒度**：方法是否过长（超过 50 行需警惕）或过短（只有一行的私有方法）
- [ ] **数据库查询**：是否有 N+1 查询、缺少索引、全表扫描
- [ ] **循环内操作**：循环内是否有数据库调用、网络请求、重量级计算

#### 深度项（HIGH/CRITICAL 风险必查）

- [ ] **抽象层次**：是否有不必要的抽象层、接口、设计模式
- [ ] **间接调用**：调用链是否过深（超过 3 层需警惕）
- [ ] **并发安全**：是否有共享变量、线程安全问题
- [ ] **资源释放**：是否有未关闭的资源（流、连接、线程池）
- [ ] **缓存使用**：是否合理使用缓存，避免重复计算/查询

### 6.6 审核结果

- 如发现问题，修复后重新审核对应项
- 所有审核项通过后，向用户展示审核结果摘要（通过项数 / 总检查项数，以及风险等级）
- 用户确认审核通过后进入第七步

## 第七步：注释提交

**目标**：写好注释，提交代码和文档。

1. **提交前验证**。根据改动范围执行对应验证：
   - 后端改动：`mvn clean compile -pl {改动涉及的模块} -am -Plocal`
   - 前端改动：`pnpm lint:eslint`
   - 验证失败 → 修复后重新验证，通过后再提交
2. **补充注释**：
   - 新增的类/方法添加 Javadoc 或行内注释说明用途
   - 复杂逻辑添加关键步骤注释
   - 不要过度注释，保持与项目现有注释密度一致
3. **提交前再次运行变更检测**（AGENTS.md 要求）：
   ```
   gitnexus_detect_changes({scope: "compare", base_ref: "master"})
   ```
   确认改动范围符合预期。
4. **提交代码**：`feat/fix/refactor/docs: 简要描述本次改动`
5. **提交计划文档**：`docs: 补充{需求概要}计划文档`
   - 文档路径：`docs/plan/YYYY-MM-DD-{需求概要}/{需求概要}.md`
   - 确保文档中的任务拆解表格已更新完成标记（✅）
6. 流程结束。

---

## 异常处理

- **用户要求回退**：如果在第 5/6 步发现方案有问题，回到第 3 步重新对齐方案，更新文档后继续。
- **用户要求终止**：随时可以终止，已完成的工作成果（特别是文档）保留在 `docs/plan/` 中。
- **步骤跳转**：正常流程不允许跳步，但用户可以明确要求回到某个步骤重新来过。
- **需求变更**：
  - 执行中小改（如调整一个字段名）→ 直接更新文档继续
  - 执行中大改（如新增模块、变更核心逻辑）→ 暂停执行，回到第二步重新对齐
